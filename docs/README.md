# OctoAcme Project Management Documentation

Welcome to the OctoAcme project management knowledge hub. This README is your starting point for understanding how OctoAcme delivers projects and where to find detailed guidance for each phase of the process.

---

## Project Management Philosophy

OctoAcme follows an iterative, milestone-driven delivery model built on a set of core principles:

- **Customer-first** — every decision prioritises customer value and usability.
- **Iterative delivery** — work is broken into small, testable increments to reduce risk and gather feedback early.
- **Clear ownership** — each project has a named Project Manager (PM) and Product Lead accountable for outcomes.
- **Data-informed decisions** — impact is measured continuously and plans are adjusted based on evidence.
- **Psychological safety** — team members are encouraged to raise concerns, share feedback, and learn openly.

---

## Project Lifecycle Phases

All OctoAcme projects move through five high-level phases:

| Phase | Description |
|---|---|
| **1. Initiation** | Define the problem statement, identify stakeholders, and agree on high-level goals and success metrics. |
| **2. Planning** | Elaborate scope, assign resources, set milestones, and map dependencies. |
| **3. Execution** | Build, test, review, and iterate in short cycles against the agreed acceptance criteria. |
| **4. Release** | Deploy to production, verify outcomes, and communicate the release to stakeholders. |
| **5. Close & Retrospective** | Capture learnings, celebrate successes, and record action items for continuous improvement. |

---

## Key Roles & Responsibilities

| Role | Core Responsibilities |
|---|---|
| **Project Manager (PM)** | Coordinates delivery, owns schedule and risk register, facilitates meetings, and drives stakeholder communication. |
| **Product Manager (PdM)** | Defines the product vision and outcomes, prioritises the backlog, and validates solutions through research and metrics. |
| **Developer** | Implements features and fixes, writes tests and documentation, participates in design and code reviews. |
| **QA / Testing** | Validates quality against acceptance criteria, identifies defects early, and signs off on releases. |
| **Stakeholders** | Provide business context and approvals, review status updates, and escalate priorities as needed. |

## Quick-Start by Persona

**If you're a Project Manager**, start with the [Project Management Overview](octoacme-project-management-overview.md) to understand OctoAcme's principles, then follow the [Project Initiation Guide](octoacme-project-initiation.md) when kicking off a new project.

**If you're a Product Manager**, read the [Project Management Overview](octoacme-project-management-overview.md) for context on how delivery works, then focus on the [Project Planning](octoacme-project-planning.md) and [Execution & Tracking](octoacme-execution-and-tracking.md) docs to understand how backlog items flow through delivery.

**If you're a Developer**, jump straight to [Execution & Tracking](octoacme-execution-and-tracking.md) for day-to-day delivery practices, and check the [Release & Deployment Guide](octoacme-release-and-deployment.md) before any release.

**If you're in QA / Testing**, start with [Execution & Tracking](octoacme-execution-and-tracking.md) to understand how work is validated, and use the [Release & Deployment Guide](octoacme-release-and-deployment.md) as a reference during release cycles.

**If you're a Stakeholder or new to the team**, the [Project Management Overview](octoacme-project-management-overview.md) and [Roles & Personas](octoacme-roles-and-personas.md) will give you the clearest picture of how OctoAcme works.

## Contributing & Updating Process Documents

These documents are living guides — they should evolve as our processes improve.

1. **Propose a change** — open a GitHub Issue using the documentation template and describe what needs updating and why.
2. **Draft your update** — edit the relevant Markdown file in the `docs/` folder on a feature branch.
3. **Submit a pull request** — link the PR to the relevant issue, add a brief description of the change, and request a review from at least one team member.
4. **Review & merge** — once approved, merge into the default branch. Update any cross-references in this README if you add or rename a document.

For significant process changes that affect multiple documents or roles, consider scheduling a brief stakeholder review before merging.
